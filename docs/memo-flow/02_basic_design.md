# 基本設計書

本書は要件定義書 `01_requirements.md` (FR-001〜FR-010) を、Task Maestro のシステム構造に落とし込んだものである。
機能系 (BD-001〜BD-010) と横断系 (BD-100〜BD-104) の 2 系統で構成する。

## システム全体像 (前提)

| 項目 | 採用 |
|------|------|
| アプリ形態 | Next.js (App Router) で構築する **ローカル起動 Web アプリ** |
| 起動方法 | `npm run dev` または `npm run start` をローカルで実行し、ブラウザで `http://localhost:3000` を開く |
| クライアント | React (Server Components + Client Components 併用)。状態管理は React Context + Server Actions |
| サーバ実装 | Next.js Route Handlers (`app/api/*/route.ts`) と Server Actions |
| 永続化 | **SQLite (`better-sqlite3`)** によるファイル DB (`data/task-maestro.sqlite`)。スキーマは BD-103 参照 |
| 認証情報 | プロジェクトルートの `.env.local` (Next.js 既定の `.env` 系列)。`.gitignore` 配下 |
| 外部 API | Slack Web API、Backlog REST API、Google Calendar API、Google Drive API、Google Docs API、OpenAI API、Claude Code CLI |
| AI 呼び出し | BD-101 (AI 抽出共通モジュール) でプロバイダを抽象化し、BD-009 (AI プロバイダ切替) で実装を選択 |
| カレンダー UI | FullCalendar (`@fullcalendar/react`) を採用。D&D・予定編集を内蔵 |
| ドラッグ&ドロップ連携 | チケット一覧側は `react-dnd` または FullCalendar 標準の外部 D&D 機能を利用 |

## BD-001: Slack メンション取り込み

- **対応する要件**: FR-001
- **区分**: 機能 (API + 画面)
- **概要**: 利用者の Slack 複数ワークスペースから、自分宛のメンションを取り込んで一覧表示し、選択して Backlog チケット化する
- **入力**:
  - 利用者の「取り込み」ボタン押下イベント
  - `.env.local` に保存された Slack ワークスペースごとの User Token (`SLACK_TOKENS_JSON` 等の JSON 配列形式)
  - 設定 (BD-100) で保持する「取り込み対象期間」(例: 過去 7 日)
- **処理概要**:
  1. 取り込みボタン押下で `POST /api/slack/mentions/sync` を呼び出す
  2. 設定済み全ワークスペースに対し `search.messages` (`from:@me` 反転して `mentions:@me`) または `conversations.history` + `users` 突合でメンションを収集
  3. 取得結果を SQLite の `slack_mention` テーブルに UPSERT (一意キー: workspace_id + ts)
  4. 未処理 (= `ticket_id` 未紐付) のメンションを一覧として返却
  5. 利用者は一覧から 1 件選択し、「チケット化」を押下
  6. チケット化時は BD-004 のチケット作成 API (`POST /api/backlog/issues`) に **元メッセージ本文 + パーマリンク + 送信者** を渡す
  7. 作成された Backlog の issue_id を `slack_mention.ticket_id` に書き戻して処理済みとする
- **出力**:
  - メンション一覧 (画面)
  - Backlog チケット (BD-004 経由で発行)
  - 処理済みフラグ更新 (SQLite)
- **連携先**: BD-004 (Backlog チケット作成・編集)、BD-008 (認証情報管理)、BD-102 (外部 API クライアント)、BD-103 (永続化)
- **データストア**: SQLite テーブル `slack_mention` (id, workspace_id, channel_id, ts, author_id, body, permalink, ticket_id, processed_at, fetched_at)
- **例外/エラー**:
  - Slack トークン無効 / スコープ不足 → 設定画面 (BD-100) に誘導するメッセージを表示
  - レートリミット (HTTP 429) → 一時待機後リトライ (最大 3 回)。連続失敗時はエラー表示
  - 部分失敗 (一部ワークスペースのみ失敗) → 成功分は反映し、失敗 WS は警告として表示
- **備考**: `users.identity` でログイン中のユーザ ID を確定する代わりに、BD-010 で手動登録した Slack ユーザ ID を「自分」とみなす

## BD-002: ミーティング議事録取り込み (Google Meet)

- **対応する要件**: FR-002
- **区分**: 機能 (API + 画面)
- **概要**: Google Calendar の終了済み予定に紐づく Google Docs 議事録を自動取得し、AI 抽出でネクストアクションをチケット候補化する
- **入力**:
  - アプリ起動時または利用者の「議事録を確認」操作
  - Google Calendar の利用者参加予定 (過去 N 日)
  - 紐づく Google Docs (添付ファイルまたは予定名一致で検索)
- **処理概要**:
  1. `GET /api/meetings/candidates` がアプリ起動中に定期 (例: 15 分間隔) で呼ばれる
  2. Google Calendar API で `timeMin = 取得済み last cursor`, `timeMax = now`, `q = ` で利用者参加予定を取得
  3. 各予定について添付ファイルまたは Drive を検索し、議事録 Doc の `documentId` を解決
  4. 議事録本文を Google Docs API で取得
  5. BD-101 (AI 抽出共通モジュール) に「議事録本文 + 利用者名 + 抽出指示プロンプト」を渡し、ネクストアクション候補配列を取得
  6. 結果を SQLite `meeting_doc` テーブルにキャッシュ (一意キー: documentId)
  7. 利用者が UI で候補を編集 / 採用 → BD-004 経由でチケット化
- **出力**:
  - 議事録候補一覧 (画面)
  - チケット候補配列 (AI 抽出結果)
  - Backlog チケット (BD-004 経由)
- **連携先**: BD-004、BD-008、BD-101 (AI 抽出)、BD-102、BD-103
- **データストア**: SQLite テーブル `meeting_doc` (id, calendar_event_id, document_id, title, occurred_at, doc_url, processed_at, candidates_json)
- **例外/エラー**:
  - 議事録が見つからない → 「議事録なし」としてマークし、再スキャン対象から除外
  - AI 抽出失敗 → 候補なし扱い。エラー詳細はログ
  - 大規模ドキュメントによるトークン超過 → 議事録を章ごとに分割して複数回抽出
- **備考**: 「自動取り込み」はアプリ起動中のみ動作する。常時稼働は要件外

## BD-003: 口頭・思いつきからの直接チケット作成

- **対応する要件**: FR-003
- **区分**: 機能 (画面 + API)
- **概要**: 「+ チケット作成」モーダルから最小入力で Backlog チケットを即時発行する
- **入力**: 利用者の手動入力 (タイトル必須、本文・プロジェクト・カテゴリ・優先度・期限・担当者は任意)
- **処理概要**:
  1. UI のヘッダーから「+ チケット作成」ボタンで BD-204 のモーダルを起動
  2. 利用者がタイトル + 所属プロジェクトを最低限入力
  3. 「作成」押下で BD-004 の `POST /api/backlog/issues` を呼び出し
  4. 作成成功で BD-005 (今日やる) リストに即追加するか確認ダイアログ
- **出力**: Backlog チケット
- **連携先**: BD-004、BD-005
- **データストア**: なし (Backlog 側に直接書き込む)
- **例外/エラー**:
  - 必須項目欠落 → クライアント側バリデーション
  - Backlog API エラー → トースト通知 + 入力内容を保持
- **備考**: モーダルからの呼び出しはダッシュボードからもチケット一覧からも可能

## BD-004: Backlog チケット表示・編集

- **対応する要件**: FR-004
- **区分**: 機能 (API + 画面)
- **概要**: 複数プロジェクトのチケットを横断取得し、一覧・編集する
- **入力**:
  - 設定で指定された Backlog プロジェクト ID 群
  - 利用者の「取り込み」ボタン押下、フィルタ操作、編集操作
- **処理概要**:
  1. `POST /api/backlog/issues/sync` で対象プロジェクト群のチケットを取得
  2. SQLite `backlog_issue` テーブルに UPSERT (キャッシュ)
  3. 一覧画面 (BD-203) で表示・フィルタ・ソート
  4. 編集時は `PATCH /api/backlog/issues/{id}` で Backlog API を呼び出し、ローカルキャッシュも更新
  5. 編集対象項目: ステータス・期限・タイトル・本文・担当者・カテゴリ・優先度・コメント追加
- **出力**: チケット一覧、編集結果
- **連携先**: BD-005 (今日やる)、BD-006 (カレンダー)、BD-008、BD-102、BD-103
- **データストア**: SQLite テーブル `backlog_issue` (id, project_id, key, summary, description, status_id, status_name, priority_id, assignee_id, due_date, updated_at, cached_at, today_flag)
- **例外/エラー**:
  - 404 (チケット削除) → ローカルキャッシュも削除
  - 422 / 409 (ステータス遷移不正) → エラー表示 + 編集前に戻す
  - 大量取得 → ページネーション (Backlog の `count=100` 上限を考慮)
- **備考**: コメント機能はチケット本文編集とは別 API (`/issues/{id}/comments`) を呼ぶ

## BD-005: 「今日やる」リスト

- **対応する要件**: FR-005
- **区分**: 機能 (画面)
- **概要**: 期限 ≤ 本日のチケット + 手動「今日やる」フラグ付きチケットをまとめて表示し、チェックでステータスを「処理中」相当に更新
- **入力**: BD-103 にキャッシュされた Backlog チケット、手動フラグ
- **処理概要**:
  1. SQLite に対し `WHERE due_date <= today OR today_flag = 1` で抽出
  2. 重複チケットは 1 件に集約
  3. 利用者が行頭のチェックボックスを押下
  4. 設定 (BD-100) の「プロジェクト × 処理中ステータス名」マッピングから該当プロジェクトの状態 ID を解決
  5. BD-004 の `PATCH /api/backlog/issues/{id}` でステータスを更新
  6. 更新成功で UI 上もチェック済みに変更
- **出力**: 「今日やる」リスト、Backlog 側ステータス更新
- **連携先**: BD-004、BD-100 (ステータスマッピング設定)、BD-103
- **データストア**: SQLite `backlog_issue.today_flag`、`settings.status_mapping`
- **例外/エラー**:
  - 「処理中」マッピング未設定のプロジェクト → エラーダイアログ + 設定画面誘導
  - ステータス更新失敗 → チェック状態を戻し、エラー表示
- **備考**: チェック解除では Backlog 側を戻さない (= 着手宣言の取り消しは手動で対応)

## BD-006: Google カレンダー連携

- **対応する要件**: FR-006
- **区分**: 機能 (画面 + API)
- **概要**: Google カレンダーのタイムラインを表示し、Backlog チケットを D&D して予定を作成・編集する
- **入力**:
  - Google Calendar 上の予定情報 (週ビュー / 日ビュー)
  - 利用者の D&D・予定編集操作
- **処理概要**:
  1. `GET /api/google/calendar/events?from=...&to=...` で表示範囲の予定を取得
  2. FullCalendar に渡してタイムライン描画
  3. 別ペインのチケット一覧からドラッグ → カレンダー上ドロップで `POST /api/google/calendar/events` を呼び出し
  4. デフォルト予定長 60 分、タイトル = チケットタイトル、説明 = チケット URL
  5. 既存予定のドラッグによる時間移動・ハンドルでの長さ変更 → `PATCH /api/google/calendar/events/{id}`
  6. 削除 → `DELETE /api/google/calendar/events/{id}`
- **出力**: カレンダー表示、Google Calendar 側の予定変更
- **連携先**: BD-004 (チケット情報)、BD-008、BD-102
- **データストア**: なし (Google Calendar を SoT とする)
- **例外/エラー**:
  - 競合更新 (Google 側で別変更あり) → 楽観ロック失敗時に再取得 + ユーザ確認
  - 予定削除済み → エラー表示 + 一覧再取得
- **備考**: カレンダー ID は設定でプライマリ固定。複数カレンダーは初期スコープ外

## BD-007: Google ドキュメント / Markdown ファイル読み込み

- **対応する要件**: FR-007
- **区分**: 機能 (画面 + API)
- **概要**: 任意の Google Docs URL またはローカル Markdown を取り込み、AI 抽出でチケット候補を生成
- **入力**: 利用者が貼り付けた Google Docs URL、またはアップロードした Markdown ファイル
- **処理概要**:
  1. 入力種別を判定 (URL or ファイル)
  2. URL の場合: Google Docs API で本文取得
  3. ファイルの場合: Next.js のアップロード API でテキスト読込
  4. BD-101 (AI 抽出) にテキスト + 抽出プロンプトを渡す
  5. 返ってきた候補配列を画面表示
  6. 採用候補は BD-004 のチケット作成 API を呼ぶ
- **出力**: チケット候補一覧、Backlog チケット
- **連携先**: BD-004、BD-008、BD-101、BD-102
- **データストア**: SQLite `imported_document` (id, source_type, source_ref, imported_at, candidates_json) — 任意 (重複取込防止用)
- **例外/エラー**:
  - Docs API でアクセス権なし → エラー表示
  - ファイルサイズ過大 → 分割処理に倒すか拒否
- **備考**: BD-002 と AI プロンプトの大枠を共有しつつ、入力種別ごとに微調整

## BD-008: 認証情報管理

- **対応する要件**: FR-008
- **区分**: 共通 (設定 + サーバ起動時検証)
- **概要**: 各 API トークン・キーを `.env.local` から読み込み、サーバ側で安全に使い回す
- **入力**: `.env.local` 内の環境変数
- **処理概要**:
  1. Next.js サーバ起動時に `process.env` を検証 (必須キー存在チェック)
  2. 必須キー欠落時はエラーログ + 設定画面 (BD-100) で警告表示
  3. クライアントへは認証情報を直接渡さず、Server Actions / Route Handlers 経由でのみ利用
- **出力**: サーバ内で参照可能な認証情報、欠落時の警告
- **連携先**: BD-001/002/004/006/007 の API 呼び出し全般、BD-100 (設定画面 UI で「設定済み」のステータスのみ表示)
- **データストア**: `.env.local` (リポジトリ管理外、`.gitignore` 必須)
- **例外/エラー**:
  - `.env.local` 未存在 → サンプル `.env.example` を案内し、起動継続 (個別 API 呼び出しはエラーになる)
- **備考**: 利用する環境変数キーの一覧は BD-100 に集約して記載

## BD-009: AI プロバイダ切替

- **対応する要件**: FR-009
- **区分**: 共通
- **概要**: AI 抽出 (BD-002 / BD-007) で利用するバックエンドを OpenAI API と Claude Code CLI から選択可能にする
- **入力**: 設定 (BD-100) の現在プロバイダ選択、対応する認証情報・実行設定
- **処理概要**:
  1. 設定値から現在プロバイダを取得
  2. `AIProviderInterface` を実装する `OpenAIProvider` / `ClaudeCodeProvider` を切替
  3. `OpenAIProvider`: `openai` SDK 経由で Chat Completions / Responses API を呼ぶ
  4. `ClaudeCodeProvider`: Node.js の `child_process.spawn` で `claude` CLI を起動し、stdin に指示 + 入力を流す
  5. 結果文字列をパースして候補配列に整形 (JSON 出力指示を含むプロンプトを使用)
- **出力**: AI 抽出結果 (チケット候補配列)
- **連携先**: BD-002、BD-007、BD-101 (この BD の利用元)
- **データストア**: 設定値 (`settings.ai_provider`)
- **例外/エラー**:
  - プロバイダ呼び出しタイムアウト → 既定で 60 秒、エラー表示
  - Claude Code CLI が PATH に無い → 設定画面で警告
  - 不正な JSON 出力 → リトライ 1 回後にエラー扱い
- **備考**: プロンプトは `prompts/extract_action_items.txt` に切り出し、プロバイダ間で共用

## BD-010: 利用者識別設定

- **対応する要件**: FR-010
- **区分**: 共通 (初期設定)
- **概要**: Slack/Google/Backlog 各サービスにおける利用者本人のユーザ ID と名前を設定保持する
- **入力**: 利用者の初期設定画面入力
- **処理概要**:
  1. 設定画面 (BD-100) の「利用者プロフィール」セクションで以下を手動登録:
     - Slack: 各ワークスペースの User ID (`Uxxxxxxx`)
     - Google: メールアドレス + 表示名
     - Backlog: User ID (数値) + 表示名
  2. SQLite `user_identity` テーブルに保存
  3. BD-001 (自分宛メンション識別)、BD-002 (議事録の自分のネクストアクション抽出)、BD-006 (カレンダー所有者照合) で参照
- **出力**: 利用者識別情報
- **連携先**: BD-001、BD-002、BD-006
- **データストア**: SQLite `user_identity` (service, identifier, display_name)
- **例外/エラー**:
  - 未設定の場合 → 該当機能を使う際に「設定が必要」と誘導
- **備考**: OAuth 認証から自動取得する案は採用せず、シンプルさを優先して手動登録のみとする

## BD-100: 設定画面

- **対応する要件**: FR-008 / FR-009 / FR-010 を統合する横断 UI
- **区分**: 画面
- **概要**: アプリ全体の設定を一画面で編集する
- **入力**: 利用者の設定操作
- **処理概要**:
  1. タブまたはセクション構成: ① 認証 (`.env.local` のキー存在チェック表示)、② AI プロバイダ、③ 利用者識別、④ Backlog プロジェクト、⑤ ステータスマッピング、⑥ Slack ワークスペース
  2. ① は表示のみ (`.env.local` ファイル編集は手動)
  3. その他は SQLite `settings` テーブルに JSON 値として保存
  4. 保存時はサーバ側で必要に応じて関連 API (Backlog のプロジェクト一覧取得など) を呼んで補完
- **出力**: 永続化された設定値
- **連携先**: BD-103、BD-009、BD-010
- **データストア**: SQLite `settings` (key, value_json)
- **例外/エラー**:
  - 必須項目未設定で保存しようとした場合 → クライアントバリデーション
- **備考**: 起動時に必須設定が未完了の場合、ダッシュボードへの遷移前にセットアップウィザードを表示する

## BD-101: AI 抽出共通モジュール

- **対応する要件**: FR-002 / FR-007 / FR-009 を支える共通基盤
- **区分**: 共通 (サーバライブラリ)
- **概要**: AI プロバイダの呼び出しと、抽出結果の JSON パースを共通化する
- **入力**: 入力テキスト (議事録本文 / ドキュメント本文)、利用者識別情報、プロンプトテンプレート、プロバイダ選択 (BD-009)
- **処理概要**:
  1. テンプレートから「ネクストアクション抽出」「タスク候補抽出」プロンプトを組み立て
  2. 必要に応じて入力テキストをチャンキング (トークン上限を超える場合)
  3. BD-009 を介してプロバイダ呼び出し
  4. 戻り値 (JSON 文字列) をパース、検証 (タイトル必須、本文任意、推奨期限 任意)
  5. 候補配列を呼び出し元に返却
- **出力**: チケット候補配列 (`{ title, body, suggested_due, suggested_assignee }[]`)
- **連携先**: BD-002、BD-007、BD-009
- **データストア**: なし
- **例外/エラー**:
  - JSON パース失敗 → リトライ 1 回後にエラー
  - 候補ゼロ → 空配列を返却 (エラーではない)
- **備考**: プロンプトはリポジトリ内 `prompts/` 配下に MD で管理

## BD-102: 外部 API クライアントアダプタ層

- **対応する要件**: FR-001 / FR-002 / FR-004 / FR-006 / FR-007 / FR-008 のすべて
- **区分**: 共通 (サーバライブラリ)
- **概要**: Slack / Backlog / Google (Calendar・Docs・Drive) / OpenAI / Claude Code CLI への呼び出しを薄くラップする
- **入力**: 各 API のリクエストパラメータ
- **処理概要**:
  1. 各サービスごとに `lib/clients/{slack,backlog,googleCalendar,googleDrive,googleDocs,openai,claudeCode}.ts` を配置
  2. 共通のエラー型 (`ExternalApiError`) でレートリミット・認証失敗・タイムアウトを正規化
  3. 認証情報は BD-008 経由で取得
- **出力**: 各 API のレスポンス (型安全)
- **連携先**: 機能系 BD すべて、BD-008
- **データストア**: なし
- **例外/エラー**:
  - 401 / 403 → 設定画面誘導
  - 429 → 指数バックオフ (最大 3 回)
  - ネットワークエラー → リトライ後、ユーザに通知
- **備考**: 各クライアントはユニットテスト可能にするためインターフェースを切り出す

## BD-103: ローカル永続化 (SQLite)

- **対応する要件**: FR-001 / FR-002 / FR-004 / FR-005 / FR-007 / FR-010 のキャッシュ・状態保持
- **区分**: 共通 (データレイヤ)
- **概要**: SQLite で取得済みデータ・設定値を保持する
- **入力**: 各 BD からの SELECT / INSERT / UPDATE
- **処理概要**:
  1. アプリ起動時に `data/task-maestro.sqlite` を開き、マイグレーション (`migrations/*.sql`) を逐次適用
  2. アクセス層は `lib/db/*` に集約し、リポジトリパターンで提供
- **出力**: 永続化されたデータ
- **テーブル一覧**:
  - `slack_mention` (BD-001)
  - `meeting_doc` (BD-002)
  - `backlog_issue` (BD-004 / BD-005)
  - `imported_document` (BD-007)
  - `user_identity` (BD-010)
  - `settings` (BD-100)
  - `migration_history`
- **例外/エラー**:
  - DB ファイル破損 → バックアップから復旧する手順を案内
- **備考**: マイグレーションは順序付きの番号付き SQL ファイルで管理

## BD-104: ダッシュボード (今日やる + カレンダー併設画面)

- **対応する要件**: FR-005 / FR-006 を統合する利用者の主画面
- **区分**: 画面
- **概要**: 左ペインに「今日やる」リスト、右ペインに Google カレンダーのタイムラインを並べ、D&D で予定化する導線を提供する
- **入力**: BD-005 のリスト、BD-006 のカレンダー
- **処理概要**:
  1. 左ペイン: BD-005 の「今日やる」コンポーネント
  2. 右ペイン: BD-006 のカレンダー (FullCalendar)
  3. 左ペインの行をドラッグして右ペインにドロップ → BD-006 の予定作成 API を呼ぶ
  4. 上部ヘッダに「+ チケット作成」(BD-003) と「取り込み」(BD-001 / BD-004 / BD-002 / BD-007 を順次起動) ボタン
- **出力**: 統合された日次ダッシュボード
- **連携先**: BD-001、BD-002、BD-003、BD-004、BD-005、BD-006、BD-007
- **データストア**: なし (各サブコンポーネントが直接 BD-103 / 外部 API を参照)
- **例外/エラー**:
  - 部分失敗 (例: カレンダー取得失敗) → 該当ペインのみエラー表示し、もう一方は機能継続 (グレースフルデグラデーション)
- **備考**: 画面遷移なしで主要操作が完結することを目指す
