# 要件定義書

## 1. 概要

本システム (以下「Task Maestro」) は、Backlog をマスタとして個人のあらゆるタスクを一元管理するためのツールである。

タスク発生源 (Slack のメンション、ミーティングの議事録、口頭の依頼、思いつき、既存の Backlog チケット) が多岐にわたるため、これらを最小限の労力でチケット化し、複数の Backlog プロジェクトを横断して「今日やるべきタスク」を可視化することを目的とする。

外部連携として Google カレンダーへの予定作成 (ドラッグ&ドロップ) や、Slack・Google ドキュメント (Meet 議事録) からの取り込みを備える。利用者は単独 (利用者本人) を想定する。

実行形態としては **ローカルで起動する Web アプリケーション (TypeScript + React/Next.js)** とし、追加サブスクリプションが発生しない範囲 (Claude Pro / OpenAI API の既存契約のみ) で構築する。

## 2. 用語定義

| 用語 | 説明 |
|------|------|
| Backlog | 株式会社ヌーラボが提供するプロジェクト管理ツール。本システムのタスクマスタ |
| チケット | Backlog における 1 件の課題。本システムにおけるタスクの最小単位 |
| ワークスペース | Slack におけるテナント。本システムでは複数ワークスペースを取り込み対象とする |
| メンション | Slack で `@<ユーザー名>` 形式で名指しされた状態 |
| ネクストアクション | ミーティング後に自分が実施すべき行動。議事録から抽出する対象 |
| タスクソース | タスクの発生源 (Slack / MTG / 口頭・思いつき / Backlog / Google Docs / Markdown ファイル) |
| AI 抽出 | OpenAI API または Claude Code CLI を用いて自由文 (議事録・メモ) からチケット候補を生成する処理 |
| AI プロバイダ | AI 抽出に用いる実装。OpenAI API / Claude Code CLI のいずれか (設定で切替可) |

## 3. 機能要件

### 3.1 Slack メンション取り込み
- **ID**: FR-001
- **概要**: 利用者が参加する複数の Slack ワークスペースから、利用者にメンションが付いたメッセージを取得し、一覧表示する。利用者は一覧から任意のメッセージを選択して Backlog チケット化できる
- **入力**: Slack API 経由のメンション情報 (メッセージ本文・送信者・チャンネル・タイムスタンプ・パーマリンク)
- **出力**: 未処理メンション一覧 (画面表示)、選択されたメンションを起点とする Backlog チケット
- **業務ルール**:
  - 取り込み対象は複数ワークスペースに跨る
  - 取り込みのトリガは **利用者が UI で「取り込み」ボタンを押した時** (手動取得)
  - チケット化済みのメンションは「処理済み」としてマークされ、再表示時に除外可能
  - チケット化時に元メッセージの本文・パーマリンク・送信者をチケットに保存する
  - 既読/未読は本システム独自管理 (Slack 側の既読状態には影響しない)

### 3.2 ミーティング議事録取り込み (Google Meet)
- **ID**: FR-002
- **概要**: Google Meet の自動議事録 (Google Docs に保存) を取り込み、AI 抽出により自分のネクストアクションを Backlog チケット候補として提示する
- **入力**: Google Calendar の終了済み予定 (利用者参加分) と、それに紐づく Google Docs 形式の議事録
- **出力**: ネクストアクション候補一覧、選択候補から作成された Backlog チケット
- **業務ルール**:
  - 議事録の検知は **Google Calendar の終了済み予定と Google Docs 議事録を自動で紐付け** て候補表示する (起動中のみ動作)
  - 紐付けに使う情報: 予定の添付ファイル (Meet メモ)、または予定タイトル・実施日と一致する Drive ドキュメント
  - AI 抽出時に利用者本人を発話者・割当先として識別する (利用者名は設定で管理)
  - 候補は利用者が編集・取捨選択した上でチケット化する
  - 元議事録の URL をチケットに参照リンクとして保存する

### 3.3 口頭・思いつきからの直接チケット作成
- **ID**: FR-003
- **概要**: タスクソースが Slack/MTG 以外の場合、利用者が手動入力で Backlog チケットを直接作成する
- **入力**: 利用者の手動入力 (タイトル・本文・プロジェクト・カテゴリ・優先度・期限・割当先など)
- **出力**: Backlog チケット
- **業務ルール**:
  - 最小入力 (タイトル + プロジェクト) で即作成可能とする
  - 残項目は後から編集可能

### 3.4 Backlog チケット表示・編集
- **ID**: FR-004
- **概要**: 複数の Backlog プロジェクトのチケットを横断して一覧・編集する
- **入力**: Backlog API 経由のチケット情報、利用者の編集操作
- **出力**: チケット一覧 (画面表示)、編集結果が反映された Backlog チケット
- **業務ルール**:
  - 対象プロジェクトは設定で複数指定可能
  - 表示項目: タイトル・状態・優先度・期限日・担当者・所属プロジェクト
  - 編集可能項目: **ステータス・期限日・タイトル・本文・担当者・カテゴリ・優先度・コメント追加**
  - 添付ファイル・カスタム属性などは編集対象外
  - 取得トリガは **利用者の「取り込み」ボタン押下時** (手動取得)

### 3.5 「今日やるタスク」管理
- **ID**: FR-005
- **概要**: 複数プロジェクトを横断して、本日着手すべきタスクを抽出・並び替え・チェックする
- **入力**: Backlog から取得したチケット群、利用者の手動選択
- **出力**: 「今日やる」リスト (画面表示)
- **業務ルール**:
  - 抽出条件: **(a) 期限日 ≤ 本日のチケットを自動抽出**、加えて **(b) 利用者が手動で「今日やる」フラグを立てたチケット** をマージ表示
  - チケット重複時は同一チケットを 1 件にまとめる
  - リスト内でチェックすると、Backlog 上のチケットステータスを **「処理中」相当** に変更する
  - 「処理中」相当のステータス名はプロジェクトごとに異なるため、設定で **プロジェクト × 状態名のマッピング** を保持する (例: プロジェクト A は「処理中」、プロジェクト B は「In Progress」)
  - チェック解除した場合のステータス戻し挙動は **戻さない** (チェック = 着手宣言と解釈し、再度未着手に戻すには手動でステータス変更)

### 3.6 Google カレンダー連携
- **ID**: FR-006
- **概要**: Google カレンダーのタイムラインを画面に表示し、Backlog チケットをドラッグ&ドロップしてカレンダー予定を作成・編集する
- **入力**: Google Calendar API 経由の予定情報、利用者の D&D 操作
- **出力**: カレンダー画面上の予定、Google カレンダーに作成・編集された予定
- **業務ルール**:
  - 予定タイトルにはチケットタイトル、説明欄にはチケット URL を埋め込む
  - 予定の長さは D&D 時点で **デフォルト 60 分**、その後ドラッグで変更可能
  - 既存予定の編集 (時間移動・長さ変更・削除) もカレンダー画面上で完結

### 3.7 Google ドキュメント / Markdown ファイル読み込み
- **ID**: FR-007
- **概要**: 任意の Google ドキュメントまたはローカルの Markdown ファイルを取り込み、AI 抽出によりチケット候補を生成する
- **入力**: Google Docs (URL) または Markdown ファイル (ローカルパス・アップロード)
- **出力**: チケット候補一覧、選択候補から作成された Backlog チケット
- **業務ルール**:
  - FR-002 と同じ AI 抽出パイプラインを共用する
  - 元ドキュメントの URL / ファイル名をチケットに保存する

### 3.8 外部連携の認証管理
- **ID**: FR-008
- **概要**: Slack / Google (Calendar・Docs) / Backlog / OpenAI の認証情報を管理し、API 呼び出しに利用する
- **入力**: 利用者による初回認証操作 (OAuth フローまたは API キー入力)
- **出力**: ローカル `.env` ファイルへの永続化、API 呼び出し時の認証ヘッダ
- **業務ルール**:
  - 認証情報は **ローカルの `.env` ファイル** に保存する。`.env` は **必ず `.gitignore` 配下** に置く
  - `.env` 例外時 (ファイル欠落・必須キー未設定) は起動時に検知し、設定画面に誘導する
  - 認証情報は **ログ出力に含めない** (マスク必須)
  - 個人ローカル利用のため、OS 資格情報マネージャ等は採用しない (シンプルさ優先)

### 3.9 AI プロバイダの切替
- **ID**: FR-009
- **概要**: AI 抽出 (FR-002 / FR-007 で使用) のバックエンドを **OpenAI API** と **Claude Code CLI** から選択できる
- **入力**: 設定画面でのプロバイダ選択、対応する認証情報 (OpenAI API キー / Claude Code 実行環境)
- **出力**: 選択されたプロバイダを用いた抽出結果 (チケット候補)
- **業務ルール**:
  - 既定プロバイダは設定で指定する。未指定時は **OpenAI API** をフォールバック既定とする
  - プロバイダごとに利用モデル名・パラメータを設定で保持する
  - Claude Code CLI 利用時はサブプロセス呼び出しとし、起動オーバーヘッドを許容する
  - プロバイダ呼び出し失敗時はエラーメッセージを表示し、もう一方のプロバイダへの手動切替を促す

### 3.10 利用者識別設定
- **ID**: FR-010
- **概要**: Slack / Google / Backlog の各サービスにおける利用者本人のユーザ ID・ユーザ名を本システム内で紐付けて管理する
- **入力**: 利用者の初期設定画面入力
- **出力**: 本人識別情報 (各サービスのユーザ ID と表示名のペア)
- **業務ルール**:
  - 初期設定画面で各サービスの ID / 名前を **手動登録** する
  - 紐付け情報は FR-001 (メンション検出)、FR-002 (議事録の発話者識別)、FR-006 (カレンダー所有者識別) で参照する

## 4. 非機能要件

| 区分 | 要件 |
|------|------|
| コスト | 既存契約 (Claude Pro プラン / OpenAI API) 以外の追加サブスクリプション・有料サービスを発生させないこと。Backlog・Slack・Google の利用は既存の無料/有料枠の範囲内で行う |
| 性能 | 各画面 (チケット一覧・カレンダー・メンション一覧・議事録候補) の **初期描画は外部 API 待ちを含めて 3 秒以内** を目標とする。3 秒を超える可能性がある場合はスケルトン表示やローディングインジケータを必ず提示する |
| セキュリティ | 認証情報を `.env` に集約し、`.gitignore` 配下に置く。ログに認証情報を出力しない。AI 抽出時に外部 API へ送信する内容を利用者が事前に確認できるオプションを持つ |
| 可用性 | 単独利用 / ローカル動作のため SLA は定義しない。外部 API 障害時はその機能のみ縮退し、他機能は継続稼働する (グレースフルデグラデーション) |
| 保守性 | 新たなタスクソース (例: メール) を追加する際、既存機能に影響を与えずに拡張できる構造を取る (AI プロバイダおよびタスクソースを差し替え可能な抽象化レイヤを設ける) |
| ユーザビリティ | キーボード主体で操作可能 (主要操作にショートカット)、最小入力でチケット化が完了する導線を提供する |
| 利用範囲 | 単独利用 (シングルユーザ前提)。マルチユーザ・権限管理は対象外 |

## 5. 制約事項

- **技術的制約**:
  - 追加サブスクリプション不可。AI 機能は Claude Pro プラン (Claude Code CLI 経由) または OpenAI API キーの枠内で実装する
  - 各外部サービスの無料 API レートリミットを前提とし、超過しない呼び出し戦略を取る (同期はユーザートリガに限定することで自然と抑制)
  - Slack ワークスペースの権限上、取得不能なメッセージ (DM 等のスコープ外) は対象外
  - 採用スタック: TypeScript + React/Next.js (フロントエンド)、Node.js API ルート / サーバアクション (バックエンド)、ローカルストレージ (詳細はストレージ選定で再検討)
- **運用上の制約**:
  - 単独利用、運用保守も利用者本人
  - 認証情報の管理責任は利用者本人

## 6. 前提条件

- 利用者は Backlog アカウントを保有しており、対象プロジェクトへのアクセス権限を持つ
- 利用者は Google アカウントを保有し、Google Calendar / Drive (Docs) を利用している
- 利用者は対象 Slack ワークスペースのメンバーであり、メンション情報を取得するための適切な権限とトークンを発行できる
- 利用者は Claude Pro プラン (Claude Code CLI が利用可能な状態) または OpenAI API キーを保有する
- 利用者のローカル PC で本アプリを起動して利用する (常時稼働環境は不要)

## 7. スコープ外

- マルチユーザ・チーム利用 (権限管理・共有・並行編集)
- Backlog 以外のタスク管理サービスへの書き出し (Jira / Asana / Notion 等)
- Slack DM・プライベートチャンネルのうち、利用者のトークンスコープでアクセスできないメッセージ
- Google Meet 以外の Web 会議ツールの議事録 (Zoom / Teams 等は今回スコープ外)
- スマートフォンアプリ・モバイル特化 UI
- メール (Gmail 等) からの取り込み
- Backlog の Wiki / ファイル / ガントチャート編集機能 (チケット系のみ対応)
- 通知機能 (デスクトップ通知・メール通知 / 未着手アラートなど)
- 常時稼働 (VPS / サーバ運用)・Webhook 受信
